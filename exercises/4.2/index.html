    Exercise 4.2 - Cross Origin Requests

    Objectives

    Learn how browser's Cross-Origin-Request rules (CORS) restricts malicious cross origin requests and how vulnerable implementations may be attacked.

    Estimated Time 15 minutes

    Requirements

    • [x] Lab VM

    Same Origin Requests

    No Hints

    Start at http://lab-4.2.sans.labs. You will see a login page. If not, click on "Logout & Reset Database" in the upper right hand corner.

    First, experiment with the donation page at "Same Origin" (you will need to log in using the username student and the password training). Secondly, after you figured out how it works, see why the "Good Cross Origin" version is broken. Finally, after you fix the "Good Cross Origin", check that the "Evil Cross Origin" is still blocked.

    Step by Step

    1. Browse to http://lab-4.2.sans.labs/login.php.

    2. Log in using the username "student" and the password "training".

    3. Open the browser's debugger (SHIFT-CTRT-I) and open the "Network" tab if it isn't already open.

    4. Enter "10" in the "Amount" text box and click "Donate". You will see a popup dialog stating that you donated a total of $110. In the debugger, you will see that "donate.php" was requested.

    5. Click on "donate.php" in the debugger. You will see the response from "donate.php". The response is a brief JSON formatted message like:

      {donations: "110", status: "success"}

    6. Click on "Headers" in the debugger's "Network" tab. You will see the request and response headers. Notice how the Origin header of the request specifies "http://lab-4.2.sans.labs" as origin. The response contains no CORS related headers. In particular, the response does not contain a "Access-Control-Allow-Origin" header.

    7. Click on the Good Cross Origin link at the top of the webpage. You will see a page that looks similar to the earlier page.

    8. Just like before, donate $10. You will not see a popup. At the bottom of the debugger's network tab, you will see a red request for "donate.php". Red indicates an error. The "Status" column indicates a "CORS error".

    9. Click on the last "donate.php" in the debugger's Network tab, and select "Headers" again. Note that the request visible in the "General" section now went to lab-4.2-api.sans.labs, not lab-4.2.sans.labs. We setup a distinct hostname for our API which is often done to better control API requests. But this request is now cross-origin.

    10. Edit the file "/etc/docker/compose/4.2/web/api/donate.php". At the top, you will find a number of commented out "header" lines. Remove the comment (delete the leading #) from the line :

      header('Access-Control-Allow-Origin: *');
      Information

      With the uncommented line, an Access-Control-Allow-Origin header will be sent from the API host indicating willingness to accept request originating from any domains (the wildcard *).

    11. The request will still fail. Look at the "Console" tab in the browser's debugger for more details. You should see a message that the wildcard can not be used if the request includes credentials. We are logged in, and credentials are transmitted as a cookie.

    12. To fix this problem, we need a more specific origin. Open "/etc/docker/compose/4.2/web/api/donate.php" again. Undo your earlier change. Instead, lets set the origin to the origin reported by the browser. Comment out the "if" block with the exception of the line we exposed earlier and comment out the wildcard host origin. After you edit, the block of code should look like the following,

      if (array_key_exists('HTTP_ORIGIN', $_SERVER) ) {
    $sOrigin = $_SERVER['HTTP_ORIGIN'];
    header('Access-Control-Allow-Origin: '.$sOrigin);
# header('Access-Control-Allow-Origin: *');
    header('Access-Control-Allow-Credentials: true'); 
}

    13. Try the donation again. It should work now. It may take a while to respond if it took you some time to make the edit as the docker container may need to restart.

      Why does it work now and not earlier?

      Access-Control-Allow-Credentials with Access-Control-Allow-Origin set to wildcard (*) will not work. A specific origin being set is allowed which is what the second edit of the file actually did. We are now specifying the exact origin to set as the Access-Control-Allow-Origin.

    14. Click on the "Evil Cross Origin". It should display a simple page with no obvious donation dialog.

    15. Go back to the last "Good Cross Origin" page and donate another $10. Note how the dialog now states that you donated over $1,000,000.

    16. Go back to the "Evil Cross Origin" page. Notice the "donate.php" request in the Network tab of the browser's debugger. If it is not visible: Reload the page.
    17. View the source of the page. You will see JavaScript sending the request donating an amount of $1,000,000.
    18. How are you going to fix this? Refer to the Netwars question for details.

    Conclusion

    CORS is an important feature allowing us to carefully restrict cross-origin requests. You may use this exercise to further experiment with CORS and see what is, and what is not, possible.
